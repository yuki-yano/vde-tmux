vde-tmux sidebar — design proposals

サイドバー改善 6項目の比較提案

Codex 向け計画書を書く前段として、6 つの改善要望それぞれに複数案を出して比較します。 モックはすべて幅 40 列の Standard ティアを想定。緑枠が推奨案です。

1 LIVE カード化 2 mode/category/repo の色 3 category 区切り線 4 Active の可視化 5 pin の強調 6 jump / preview

Item 1

LIVE を罫線で囲って card 化

現状の LIVE は「見出し行+本文」がツリーと地続きで、FLEET との境界が曖昧。 全体を罫線で囲んで card として独立させたい。

案 A — 角丸フルボーダー(タイトル埋め込み)

╭╴LIVE tail · %12 ─────────────╮
│ ✓ cargo test … 373 passed  │
│ ➜ clippy clean             │
╰──────────────────────────────╯
  • ユーザー要望どおりの card 感が最も強い
  • タイトルを枠上辺に埋め込むので追加行は上下 +1 行ずつのみ
  • ratatui の Block::bordered().border_type(Rounded).title(...) でほぼ標準機能
  • 左右 1 列ずつ本文幅が減る(40 列中 4 列がフレーム+パディング)
  • ANSI 長行のトリミング位置の再計算が必要

案 B — 上下罫線のみ(幅を食わない card)

─╴LIVE tail · %12 ──────────────
 ✓ cargo test … 373 passed
 ➜ clippy clean
────────────────────────────────
  • 本文幅の損失ゼロ。狭幅サイドバーと相性が良い
  • 実装が最軽量(見出し行と末尾に rule を描くだけ)
  • 「囲う」要望からは半歩後退。card というよりセクション区切り

案 C — フルボーダー+LIVE 色の枠

┌╴LIVE╶tail · %12 ───────────┐
│ ✓ cargo test … 373 passed  │
└──────────────────────────────┘
  • LIVE 色(magenta)の枠で存在感が最大
  • 常時表示される帯としては主張が強く、バッジの状態色より目立ってしまう
  • 枠色がノイズになり「詰まりは赤」の視線誘導を弱める

推奨: 案 A。枠線は marker 色(控えめグレー)、タイトルの LIVE だけ live 色で従来の配色を維持。 幅 24 列未満(Dense/Micro)では LIVE 自体が実用にならないため、狭幅時は案 B の上下罫線へ自動フォールバックすると割り切りが良い。

実装影響: tui.rs render_live_lines を Block ベースに変更。sidebar.live.border(full/rule/none)を config に追加するかは計画書で判断。

Item 2

view mode 名 / category 名 / repo 名の色分け

現状は 3 つとも repo 色(青+BOLD)で描画されており、ヘッダーの category とツリーの category 行、repo 行の区別が付かない。

案 A — 3役3色(色だけで分離)

 category · ≡6 ▲1 ●2 ✓1 ○2
 ▾ dev
   ▾ vde-tmux  main +2
     ● claude  sidebar polish
  • mode=ラベンダー / category=ピーチ / repo=青。statusline の category ピル(FAB387)と色が揃い、ツール全体で「category=ピーチ」が通る
  • 実装が単純(theme に 2 色追加して参照を差し替え)
  • 色覚特性によっては青/ラベンダーの距離が近い

案 B — 色+接頭グリフの二重符号化

 ≣ category · ≡6 ▲1 ●2 ✓1 ○2
 ▾ ◆ dev
   ▾ vde-tmux  main +2
     ● claude  sidebar polish
  • 案 A の色分けに加え、mode に ≣、category に ◆ を付けて形でも区別。色に頼らない
  • repo は無印+git バッジ、と役割ごとの見た目文法が確立する
  • グリフ 2 文字ぶん幅を消費(狭幅ティアでは省略が必要)

案 C — 色は据え置き、形だけで分離

 [category] · ≡6 ▲1 ●2
 ▾ dev
   ▾ vde-tmux  main +2
  • theme への色追加が不要で config 互換の心配がない
  • 同色のままなので一瞥での区別は改善が小さい
  • 下線・括弧は端末フォントによって視認性がばらつく

推奨: 案 B。色(3役3色)と形(グリフ)の二重符号化で、Dense 以下の狭幅ではグリフを落として色のみ運用。 category のピーチは statusline との一貫性が取れるのが決め手。

実装影響: theme に colors.category と colors.header_mode を追加(既定: ピーチ / ラベンダー)。render.rs row_style の Category 分岐を repo 色から分離、mode_segment_style を新色へ。

Item 3

category 間の区切り線

ByCategory 表示で category グループの境目が行間だけでは判別しづらい。横線で区切りたい。

案 A — category 行を rule 一体型に

 ▾ ◆ dev ──────────────── ▲1
   ▾ vde-tmux  main
     ● claude  sidebar polish
 ▾ ◆ infra ──────────────────
   ▸ dotfiles  main
  • 追加行ゼロで区切りが入る(縦空間を消費しない)
  • category 行自体が見出し化し、Item 2 の色分けと相乗効果
  • 行クリック(開閉)の当たり判定が線の分だけ広がる
  • 右端の ▲n と rule の間隔計算がやや面倒

案 B — グループ間に独立した罫線行

   ▸ dotfiles  main
 ────────────────────────────────
 ▾ ◆ infra
   ▸ k8s-conf  main
  • 最も明快な分断。実装も単純(Zone 的な非選択行を挿入)
  • category 数 −1 行ぶん縦空間を消費。agent が増えるほど痛い
  • 罫線行がカーソル移動のスキップ対象になり、選択ロジックに手が入る

案 C — 空行スペーサーのみ

   ▸ dotfiles  main

 ▾ ◆ infra
  • 実装最小・視覚ノイズ最小
  • 「線で区切りたい」という要望を満たさない
  • 狭幅では空行が境界として読み取りにくい

推奨: 案 A。縦空間を使わずに要望を満たす。ByRepo 表示でも同じ描画を repo 行に適用するかは 計画書で要検討(repo 行は git バッジがあるため rule は短くなる)。TRIAGE 見出しの ▍ とはスタイルが被らない。

実装影響: render.rs の category 行描画で残余幅を ─ パディング。新規行種別は不要。

Item 4

現在 Active な category / repo / agent の可視化

いま自分がいる tmux セッション(=Active な agent とその系譜)がツリー上で見分けられない。 選択ハイライト(カーソル)とは別の「現在地」表示が欲しい。

案 A — Active 行に薄い背景色

 ▾ ◆ dev                        
   ▾ vde-tmux  main             
     ● claude  sidebar polish    
     ✓ codex   review done
     ○ codex   (カーソル行)
  • 要望どおりの「薄い背景」。面で見えるので現在地が最も分かりやすい
  • 選択背景(237)と Active 背景の 2 種類の面ができ、暗色テーマで濁りやすい
  • カーソルが Active 行に乗ったときの合成ルールを決める必要がある

案 B — 左端インジケータバー(系譜を縦に貫く)

▎▾ ◆ dev
▎  ▾ vde-tmux  main
▎    ● claude  sidebar polish
     ✓ codex   review done
     ○ codex   (カーソル行)
  • category → repo → agent の系譜が 1 本の縦バーで繋がって見える
  • 幅 1 列で済み、選択背景・状態色のどちらとも干渉しない
  • Micro/Dense ティアでもそのまま使える
  • 「背景で塗る」案ほどの面的な強さはない

案 C — Active agent のみ背景+祖先はマーク

 ▾ ◆ dev ◂
   ▾ vde-tmux  main ◂
     ● claude  sidebar polish    
     ✓ codex   review done
  • agent は面、祖先は点、と情報量に応じた強弱が付く
  • 右端の ◂ が ▲n や経過時間ラベルと同じ領域を奪い合う
  • 符号が 2 種類になり文法として覚えにくい

推奨: 案 B を基本、Active な agent 行にだけ案 A の薄背景を重ねる折衷。 縦バーで系譜、薄背景で本体、カーソルは従来の 237+BOLD が最前面。3 層が重なっても 「バー(1列)+背景(面)+BOLD」で干渉しない。

実装影響が最大の項目。daemon が「現在の client のアクティブ pane/session」を知る必要がある (poll で tmux display-message -p 相当を取得し、SidebarRow に active: bool を追加、祖先へ伝播)。 theme に colors.active_bg / colors.active_bar を追加。複数 client 接続時の扱い(最後にフォーカスした client 基準)は計画書で明記が必要。

Item 5

pin の強調

現状の pin マーカーは DarkGray の · 1 文字で、pin したこと自体を見失う。

案 A — グリフ変更+アクセント色

 ✦● codex  4m · statusline fix  2m
  ○ claude
  • 1 文字のままレイアウト不変で、色(ラベンダー)+形(✦)の両方で目立つ
  • pin の meta 1 行(fisheye 中段)にも同色を使えば「pin されている塊」が読める
  • グリフのレンダリングがフォント依存(Nerd Font 環境なら問題なし。代替 ◆ を既定にする手もある)

案 B — 右端に pin ラベル

 ·● codex  4m · fix   ⟟pin 2m
  • 文字で明示するので誤読がない
  • 右カラムは経過時間・wait 等の状態ラベルの定位置で、幅の奪い合いになる
  • 4 文字の常時消費は狭幅で致命的

案 C — pin 行に薄背景

 ·● codex  4m · statusline fix  
  ○ claude
  • 面で塗るので視認性は最強
  • Item 4 の Active 背景と面表現が衝突し、「塗られている=どっち?」になる
  • pin は複数あり得るため、画面が塗りだらけになる

推奨: 案 A。面表現は Item 4(Active)に譲り、pin は「点だが強い色形」で分担する。 マーカー色は theme の新キー colors.pin(既定ラベンダー)にし、pin 中の meta 行の先頭マーカーも同色に。

実装影響: render.rs の pin マーカー描画(グリフ・色)の差し替えのみ。グリフは badge.glyphs 系と同様に config 化。

Item 6

jump 横に preview を配置+detail クリックの意味変更

preview の入口が「detail 行シングルクリック(+250ms のダブルクリック判定)」で分かりにくい。 preview を jump の隣にボタンとして出し、detail(グレー文字)クリックは展開トグルに変えたい。

案 A — 1 行 2 ボタンのアクション行

   ● claude  sidebar polish     4m
     prompt: LIVE の色を理想形に…
     session: dotfiles / pane: %12
     ↗ jump   ⌕ preview
  • 要望どおり並列配置。クリック先が明示的で、ダブルクリック判定(250ms)を全廃できる
  • detail 行クリック=展開トグル、と各行の意味が 1 つに定まる
  • クリックの当たり判定を「行」から「行+列範囲」に拡張する必要がある(MouseEvent には列座標があるため実現可能)

案 B — jump / preview を 2 行に分ける

     session: dotfiles / pane: %12
     ↗ jump
     ⌕ preview
  • 既存の「行単位クリック」の仕組みのまま実装できる(列 hit-test 不要)
  • fisheye のフル展開が 1 行伸びる。展開行数はすでに 5〜7 行あり縦に重い
  • 「jump の横に」という要望と異なる

案 C — アクション行+キー割当も明示

     ↗ jump⏎   ⌕ previewp
  • 案 A に加えキーヒント(Enter/p)を添えて学習コストを下げる
  • ヒント文字が常時表示されるとノイズ。フッターの早見と役割が重複

推奨: 案 A。合わせて挙動を次のとおり整理する:

  • ↗ jump クリック / Enter → ペインへジャンプ(従来どおり)
  • ⌕ preview クリック / p → フローティング popup でプレビュー
  • detail 行(グレー文字)クリック → その chat の手動展開のトグル(選択を外しても開いたまま/畳む)。プレビューは開かない
  • detail 行のダブルクリック判定(250ms 待ち)は廃止 → シングルクリックの反応が即時になる副次効果

実装影響: ClickTracker に列範囲の hit-test を追加(行種別 Jump の行内に 2 つのアクション領域)。 行種別 Jump を Actions に改名するか、jump 行に preview スパンを足すかは計画書で決定。 daemon 側 toggle:<row_id> の detail 行対応(親 chat の manual expand をトグル)を追加。
