構造は良い、視覚設計に伸びしろがある

daemon push によるリアルタイム更新、rollup の min() 集約、attention フィルタ、rail モードなど、データ設計は既に強い。課題は「文字列の並べ方」に集中している。

  • 余白ゼロ — 全行が左端に密着し、pane の境界と文字が接触する。右端も同様。
  • 絵文字バッジの幅問題 — 🔴🟡は端末・フォントにより1〜2セル幅が揺れ、列が崩れる。彩度も高くノイズになる。
  • 右端が揃わない — [running:2] や 2/5 がラベル直後に付き、視線の第2軸(状態・数値)が作れていない。
  • truncate がぶつ切り — truncate_width は省略記号なしで chars を切るため、切れたことすら分からない。
  • Detail 展開が重い — 1 agent の詳細に9行消費し、40列 × 実用高さでは他が見えなくなる。
  • ヘッダーが状態表示のみ — repo all は固定幅パディングの残骸が見え、クリック可能であることが伝わらない。
  • フッターなし — j/k/Tab/v/p などのキーが発見できない。

01先行事例 — herdr と tmux-agent-sidebar

同じ問題(複数 AI エージェントの tmux 監視)に対する2つの対照的な回答。herdr は「浅く・少なく・一目で」、tmux-agent-sidebar は「深く・多く・行内に全部」。vde-tmux の現行は両者の中間にいる。

 llm-proxy  master
 qmp        autoresearch/qmp-ra…
 agents                    all
 herdr      working · claude  ◉
 explore    idle · opencode   ✓
 llm-proxy  blocked · claude  ●
 qmp        done · codex
状態4値(blocked/working/done/idle)。1 agent = 1行に徹する
tmux-agent-sidebar — リッチカード
 ≡6 ●1 ◎1 ◐2 ○2 ✕0

 ● claude  main +        13m
   ❯ fix sidebar flicker
   ▷ Found it. The rollup rec…
   ✔✔◼◻◻ 2/5
   ├ Explore #a1b2
   └ Bash    #dead
 ◐ codex   main  permission
   ❯ review PR #42
状態5値 + 優先順位解決。1 agent = 2〜7行の可変カード

herdr から学べること

  • 2段分離:「どこで(spaces)」と「誰が何を(agents)」を別セクションにし、ツリーの折りたたみ操作を消している。→ 案B の骨格。
  • 状態は4値に畳む:blocked / working / done / idle。人間の次の行動(対応する・待つ・確認する・無視する)に1:1対応する粒度。vde-tmux の BadgeState と同型。
  • 行は名前+状態 · agent種別のみ。prompt すら出さない潔さで密度を確保。

tmux-agent-sidebar から学べること

  • フィルタバー = 状態サマリ:最上部に ≡6 ●1 ◎1 ◐2 ○2 ✕0 と「アイコン+ライブ件数」を並べ、それ自体がクリック/h/l/Tab で切り替わるフィルタになっている。表示と操作の統合として秀逸。その下に repo セレクタ ▾(r キー)。
  • 単幅グリフ5種:● running / ◎ background / ◐ waiting / ○ idle / ✕ error。形状差があり色に依存しない。競合時は running > permission > background > waiting > idle の優先順位で1つに解決 — vde-tmux の RollupLevel min() と同じ発想。
  • task 進捗のグリフ化:✔✔◼◻◻ 2/5。数字だけより進行の形が見える。vde-tmux は parse_tasks 済みなので移植が容易。
  • ▷ レスポンスプレビュー:agent の最後の発言1行を行内に出す。「jump する価値があるか」の判断材料として prompt より強いことが多い。
  • worktree 統合:sidebar から worktree+agent を spawn、1キーで window/worktree/branch を teardown。vde-tmux には vde-worktree があるので同じ統合が狙える。
  • デスクトップ通知:完了・permission・エラー時にネイティブ通知。サイドバーが見えていない時間をカバーする。

02前提:状態グリフの共通刷新

4案すべてに共通する前提として、絵文字バッジを「単幅グリフ + ANSI色」に置き換える。色だけでなく形も変えることで、色覚特性やモノクロ端末でも判別できる。BadgeState 4値との対応:

▲
blocked🔴 ← permission / error / waiting
●
working🟡 ← running / background
✓
done🔵 ← idle + unread
○
idle🟢 ← idle(既読)

意味論の変更提案:現行は working=🟡黄・idle=🟢緑だが、「進行中=緑(健全)」「要対応=赤」「完了未読=シアン」「静止=無彩色」に再割り当てする。緊急度と彩度が一致し、視線が自然に blocked → working の順に流れる。現行の黄=working を維持したい場合は BadgeGlyphs 設定で色だけ差し替え可能な構造は既にあるので、既定値の変更として提案する。

working の ● は daemon push のタイミングでスピナー(◐◓◑◒)にすると「生きている」ことが伝わるが、更新頻度依存なのでオプション扱いが妥当。

代替案(tas 型5値直接表示):tmux-agent-sidebar のように BadgeState への畳み込みをやめ、RollupLevel 寄りの5グリフ(✕ error / ● running / ◐ waiting·permission / ◎ background / ○ idle)を直接出す手もある。情報は増えるが、「blocked をまとめて赤1色で叩きつける」畳み込みの強さは失う。attention 駆動の運用なら4値、状態の内訳まで常時見たいなら5値。どちらでも BadgeGlyphs 設定の拡張で吸収できる。

変更点

  • 全行に左右1列の padding。行頭の選択マーカー "> " は廃止し、選択は背景色バーのみで表現(現行も bg は付くので二重表現を解消)。
  • 右端整列カラムを導入。Repo 行は右端に attention 件数(▲1)、Chat 行は右端に状態略語(perm/wait)または経過時間。[running:2] 形式は廃止。
  • Detail 9行 → 選択行の inline meta 1行。選択中の Chat 行の直下にだけ 13m · task 2/5 · sub 2 ⏎jump を自動表示。Space の明示展開は「フル詳細(現行 Detail)」として残す。
  • ヘッダーを2ゾーン化:左=モード・フィルタ(クリック可能)、右=状態サマリ(▲2 ●1 ○2)。固定幅パディングをやめ、· 区切りに。
  • フッター1行にキーヒント。高さが閾値未満なら自動で消す。
  • truncate は … 付き(unicode-width ベースで表示幅を計測)。
  • 展開マーカーを v/> → ▾/▸ に。ASCII fallback は設定で。

Pros

  • state / tree / daemon に一切手を入れず render.rs 中心で完結
  • 現行ユーザーの操作モデルが崩れない
  • inline meta により Enter 前に必要十分な情報が得られる

Cons

  • repo が増えるとツリーの折りたたみ操作が必要な点は変わらない
  • attention は依然「各 repo の中」に散る(ソートとバッジで緩和)
実装メモ:render.rs の render_row_text/build_header_layout の書き換え + truncate_width の unicode-width 化 + フッター行追加(tui.rs の layout 分割を3分割に)。既存テストの期待文字列更新が主作業。規模感 小。

BTwo-Deck — herdr 型の2段サイドバー

向くケース:リポジトリ(space)を跨いだ作業が常態 / 「どの repo にいるか」と「agent の状態」を別のレイヤーとして扱いたい

案B · 40 cols
 SPACES                             3
 vde-tmux       main ↑2          ▲ 
 obsidian-sync  main             ▲
 dotfiles       main             ·
 ────────────────────────────────────
 AGENTS · vde-tmux              all
 ● claude   working · 13m
   └ fix sidebar flicker      2/5
 ▲ codex    permission
   └ review PR #42
 ○ claude   idle · %15
 ────────────────────────────────────
 ⇥デッキ切替 ⏎jump a:全repo表示
上段で repo を選ぶと下段がその repo の agent に絞られる
案B · AGENTS デッキで all 表示
 SPACES                             3
 vde-tmux       main ↑2          ▲
 obsidian-sync  main             ▲
 ────────────────────────────────────
 AGENTS · all             attention
 ▲ codex    permission     vde-tmux 
   └ review PR #42
 ▲ codex    waiting   obsidian-sync
   └ sync vault
 ● claude   working · 13m vde-tmux
   └ fix sidebar flicker      2/5

構成

  • 上段 SPACES:repo 一覧。各行は「名前 / branch ↑↓ / 状態ロールアップ(最悪値1グリフ)」。高さは repo 数 + ヘッダー(最大でウィンドウの1/3、超えたらスクロール)。
  • 下段 AGENTS:選択 repo の agent をフラット表示。各 agent は2行(1行目=名前+状態+経過、2行目=prompt+タスク進捗)。a で全 repo 横断表示に切り替え、その場合は右端に repo 名を表示。
  • Tab でデッキ間フォーカス移動(現行の Tab=フィルタは f へ移設、またはヘッダークリック)。j/k は現在のデッキ内で動く。
  • 区切りは罫線1本。デッキ見出しは SPACES/AGENTS の小文字ラベルを dim で。

herdr との差分

herdr は spaces=サーバー上のワークスペースだが、vde-tmux では既にある (category, repo) グルーピングを上段に写像するだけでよい。カテゴリを使う場合は上段を category、下段を repo+agent にする変形も可能。

Pros

  • 「repo の選択」と「agent の監視」が別ゾーンになり、折りたたみ操作が消える
  • 上段が常時見えるダッシュボードとして機能(全 repo の最悪状態が一望)
  • ネスト深度が最大1になり、40列に対してインデント消費が最小

Cons

  • 2フォーカスモデルの導入で state / 入力 / クリック処理の改修が最大
  • repo が1〜2個の場合、上段が空間の無駄になる(repo 1個なら上段自動非表示で緩和)
  • 下段2行/agent なので一覧密度は案Dに劣る
実装メモ:SidebarState に focused_deck と selected_space を追加、tree.rs は2つの行リストを返す形に分割、tui.rs の描画を3レイアウト(上段/罫線/下段)に。クリックのデッキ判定も必要。規模感 大。

CAttention Inbox — 状態バケツでの再グルーピング

向くケース:agent 数が多く「次に自分が触るべきもの」の判断が主目的 / attention フィルタを常用している

案C · 40 cols
 inbox                     6 agents
 ▍ATTENTION 2
 ▲ codex  review PR #42        perm 
   vde-tmux · 2m
 ▲ codex  sync vault           wait
   obsidian-sync · 8m
 ▍RUNNING 1
 ● claude fix sidebar flicker…  13m
   vde-tmux · task 2/5 · sub 2
 ▍QUIET 3
 ✓ claude update readme    obsidian
 ○ claude —                    vde
 ────────────────────────────────────
 ⏎jump n:次のattention ␣QUIET開閉
ATTENTION が常に最上部。QUIET はデフォルト折りたたみも可

構成

  • ツリーの第一軸を repo から 状態バケツに変える:ATTENTION(blocked)→ RUNNING(working)→ QUIET(done+idle)。BadgeState の Ord がそのままセクション順になる。
  • 各 agent は2行:1行目=グリフ+agent+prompt+状態略語(右端)、2行目=repo・経過・進捗を dim で。
  • セクション見出しは左端の色付き縦線 ▍ + ラベル + 件数。件数が0のセクションは見出しごと非表示。
  • QUIET はデフォルト折りたたみ(1行のサマリだけ)にすると、実質「attention フィルタが構造に昇格した」状態になる。
  • repo 情報は2行目に降格。repo で絞りたいときは既存の ByRepo モードに v で切り替えればよい —— この案は ViewMode の4つ目(ByStatus)として追加するのが自然。

Pros

  • 「今すぐ対応すべきもの」が構造的に最上部へ集まり、フィルタ切り替え自体が不要になる
  • 状態変化で行が上下に動くため、変化への気づきが生まれる
  • ViewMode 追加として実装でき、既存モードと共存(1/2/3/4 で直接切替)

Cons

  • repo の文脈が弱く、同名 agent が並ぶと2行目を読む必要がある
  • 状態変化のたびに行が移動するので、選択追従(id ベースの selection は既にあるので致命的ではない)と視線の安定性に配慮が要る
実装メモ:tree.rs に build_rows_by_status を追加し、ViewMode::ByStatus を state.rs に追加。セクション見出しは SidebarRowKind::Category を流用可能。render は案Aの刷新に乗る。規模感 中。

DDense Monitor — 1行1エージェントの高密度表示

向くケース:agent 10体以上を並走させる / サイドバー幅を30列以下に切り詰めたい / rail と通常表示の中間が欲しい

案D · 40 cols
 agents 6  ▲2 ●1 ✓1 ○2     [all]
 ▲ codex   vde review PR #42  perm
 ▲ codex   obs sync vault     wait
 ● claude  vde fix sidebar f… 13m 
 ✓ claude  obs update readme    ✓
 ○ claude  vde —                ·
 ○ claude  dot —                ·
 ────────────────────────────────────
 ● claude · vde-tmux · %12 · 13m36s
 task 2/5 · sub: Explore, general…
下部2行はステータスペイン:選択行の詳細が常時ここに出る
案D · width 24(狭幅でも成立)
 6 ▲2 ●1 ✓1 ○2
 ▲ codex  vde perm
 ▲ codex  obs wait
 ● claude vde 13m 
 ✓ claude obs   ✓
 ○ claude vde   ·

構成

  • グルーピングを完全に捨て、緊急度ソートの1行リストにする(現行 Flat モードの強化版)。ソートは既存の compare_agent_panes(attention → rollup → pane_id)がそのまま使える。
  • 各行は固定カラム:グリフ · agent名(7字) · repo略号(3字) · prompt(可変・…) · 右端状態。列が揃うことで表として走査できる。
  • repo 略号は repo 名の頭3字(衝突時は4字)。dim 表示で主張させない。
  • 詳細はツリー展開ではなく下部ステータスペイン(2行固定)に常時表示。選択を動かすだけで詳細が流れる —— Detail 行・Space 開閉が不要になる。
  • ヘッダーは件数サマリ(▲2 ●1 ✓1 ○2)。これは rail モードの表示(後述)と同じ語彙。

Pros

  • 密度最大:高さ24行で約20 agent。並走数が多いほど効く
  • 幅24列まで自然に縮退でき、rail(2列)との中間形態になる
  • 展開・折りたたみの概念が消え、操作が j/k/Enter だけになる

Cons

  • repo の文脈が3字略号だけになり、repo 数が多いと苦しい
  • prompt の表示幅が最も狭い(40列で約14字)
  • ステータスペイン分の高さ(2〜3行)を常時消費
実装メモ:ViewMode::Flat の render を専用パスに分岐し、固定カラム整形関数を追加。ステータスペインは tui.rs のレイアウトを List+Paragraph の2段にする。規模感 中。

05横断デザイン指針 — 余白・整列・truncate・rail

どの案を選んでも適用すべき、40列のテキスト UI としての設計原則。

余白(margin / padding)

  • 左右1列ずつの padding を全行に。tmux の pane border と文字が接触しないだけで別物に見える。幅40列なら実効38列。rail(2列)時のみ padding なし。
  • 選択ハイライトは padding 込みのフル幅で塗る。行頭 "> " マーカーとの二重表現はやめ、bg + bold に一本化。マーカーを残すなら bg を消す(どちらか一方)。
  • セクション間は罫線より空行を優先。罫線(─)は「デッキ境界」など構造的に強い境界だけに使い、repo 間のような弱い境界は空行または見出し色で分ける。線が多いと40列では柵に見える。
  • ヘッダー・フッターは本文と1行分の呼吸(空行 or 罫線)を挟む。高さが20行未満のときはフッター→ヘッダーサマリの順に自動で畳む。
  • インデントは2スペース、ネストは最大2段まで。40列でインデント3段(6列)は prompt 表示幅を15%奪う。案B/C/D がネストを浅くしているのはこのため。

整列 — 右端カラムという第2の読み軸

左端の「グリフ列」と右端の「状態・数値列」を固定カラムにすると、中央の可変テキストを読まずに縦スキャンだけで全体を把握できる。[running:2] のようにラベル直後へ付ける現行方式は、この縦の軸を壊している。経過時間・件数などの数値は右端揃え + tabular に。

truncate

  • 省略記号 … を必ず付ける(1文字分を確保して切る)。
  • chars().count() ではなく 表示幅(unicode-width)で計測する。CJK・絵文字を含む prompt で右端カラムが崩れる現行の潜在バグもこれで直る。
  • 切る位置は「中央の可変テキストのみ」。右端カラムとグリフは絶対に切らない。

07推奨と進め方

推奨:案A を基盤として先行実装し、tas 由来のフィルタバー(icon+count)を取り込み、案C(ByStatus)を4つ目の ViewMode として重ねる。案B(2段デッキ)は state・入力系の改修が最大、案E(リッチカード)は密度と実装コストの点で全面採用は見送り、それぞれ「次段の選択肢」「要素の取り込み元」として扱う。

  • Phase 1(案Aの土台):グリフ刷新・左右 padding・右端カラム・truncate 改善・フッター、そしてフィルタバーの icon+count 化。render.rs 中心、既存テストの期待値更新が主作業。ここまでで見た目の課題は8割解消する。
  • Phase 2(inline meta + 機能):選択行 inline meta、n ジャンプ、既読管理、状態変化フラッシュ、task グリフ(hook 拡張が許すなら)。
  • Phase 3(案C):ViewMode::ByStatus を追加し 4 キーに割当。attention 運用が主なら既定モードにする。
  • Phase 4(worktree 連携):vde-worktree との接続(spawn / teardown / + マーカー)。tas が実証した「サイドバー = agent の起点」への拡張。
  • Phase 5(必要なら案B/D/E):並走 agent 数が10体超なら案D の密度、repo 横断が常態なら案B の2段化、深い監視が主なら案E の ▷ プレビュー(daemon 拡張)を再評価する。

いずれの Phase も BadgeGlyphs / SidebarColorsConfig の既存カスタマイズ機構を壊さない範囲で「既定値の変更」として実装できるため、絵文字バッジ派のユーザーは設定で現行表示に戻せる。

原理① モードの廃止 — 状況が表示を決める

Flat/ByRepo/ByCategory や rail の切替キーをなくし、幅・高さ・agent 数・attention 有無から表示密度を導出する。幅を広げれば勝手にリッチに、狭めれば勝手に密になる(左表)。TRIAGE ゾーンは attention が0なら消滅し、FLEET が全域を使う。repo が1つなら repo 見出しも消える。「ユーザーが表示を管理する」仕事自体を削除するのが最大の UX 改善で、5案を「選ぶ」必要もなくなる — 全部が同一 UI の断面になる。

スコープ:素の状態は全 category・全 session 横断(現行 daemon の収集範囲と同じ)。絞り込みはフィルタバー右端の scope ▾(category → repo の2段選択)で行い、— なら全横断。ただし TRIAGE だけはスコープを貫通して常にグローバル — 絞り込み中に permission 待ちを見落とすのが最悪ケースなので、スコープ外の attention も行内に codex · work/vde-tmux と出所付きで表示する。category 見出しも自動適応の対象で、category が1つなら非表示、複数あるときだけ repo 見出しに dim で前置(▾ work/vde-tmux · main)し、ネストは深くしない。

原理② Fisheye — 選択だけがリッチ、ただし pin は残す

密度(案D)と情報量(案E)のトレードオフは、全行を同じ高さにする前提から生じる。選択行だけを2〜4行(▷レスポンス・task グリフ・subagent)に展開し、非選択行は1行に保てば、一覧性と詳細性が同時に成立する。j/k で舐めるだけで各 agent の詳細が流れていく。ratatui は毎フレーム全再描画なので行高の動的変化に技術的障害はない。

手動の開閉(Space)は廃止せず「pin」に意味を変えて残す。選択は一時的な視線、開閉は持続的な関心であり、別物だからだ。「この2体の進捗を見張りながら選択は別の agent を触る」が fisheye 単独では成立しない。行高は3段階:

  • 選択行 — フル展開(▷・task・subagent、2〜4行)。選択が離れたら自動で畳まる。
  • pin 行(Space でトグル、左端に · 等の pin 印) — 中展開(task グリフ+▷ の2行)を選択がどこにあっても維持。
  • その他 — 常に1行。

pin をフル展開にしないのは密度防衛のため。pin だらけにすると fisheye の前提が崩れるので、中展開に制限した上で、高さが不足したら「最後に触った pin から順に1行へ縮退」する(pin 自体は保持し、余裕が戻れば再展開)。フル詳細が2体以上で常時必要なら、それはもう LIVE ペインの分割か preview の仕事と割り切る。なお repo 見出しの開閉(グループの折りたたみ)は現行どおり残る — こちらは関心の管理そのものなので fisheye と競合しない。

原理③ サイドバーから出ない — 行内アクションと LIVE ペイン

  • permission への行内応答:TRIAGE の permission 行で y/n を押すと、対象 pane へ send-keys で応答を送る。何を許可するかはコマンド行($ cargo sqlx migrate run)と d(diff/コンテキストを floating preview)で確認できる。jump → 応答 → 戻るという往復が1キーになる。サイドバーの利用時間の大半は permission 対応なので、これが理想形の中核。
  • LIVE ペイン:選択 agent の pane 末尾2〜3行を capture-pane で常時ストリーム表示。「jump する価値があるか」を prompt(過去)でも ▷(要約)でもなく現在の画面そのもので判断できる。
  • worktree spawn/teardown(w/x)と / 検索、n/N attention 巡回で、agent のライフサイクル全体がサイドバーで完結する。

時間軸の補完

  • 状態遷移フラッシュ:変化行を1.5秒 reverse video。
  • イベントログ(e でトグル):LIVE ペインを「2m前 codex → permission」の遷移フィードに切替。離席中に何が起きたかを取り戻せる。
  • デスクトップ通知 + rail 点滅:サイドバーが見えていない/畳んでいる時間をカバー。
  • 経過時間は毎分 tick で live 更新、working グリフはスピナー。

成立させるための代償

  • daemon 拡張:▷用の last_response hook、capture-pane ストリーム、遷移イベントバッファ
  • send-keys 応答の安全装置(対象 pane の状態を再確認してから送る)
  • 自動変形の境界値はユーザー調整可能にしないと「勝手に変わる」不快感が出る

それでも残る限界

  • 行高が動く fisheye は視線の安定性と引き換え(選択移動で下が押し流れる)→ 展開は viewport 内に収める scroll 制御が必須
  • permission の行内応答は「読まずに y を叩く」誘惑を生む。コマンド行の常時表示は外せない
段階導入との関係:この理想形は案A(視覚基盤)→ フィルタバー → TRIAGE 分離(案C)→ LIVE ペイン(案D の発展)→ fisheye → 行内応答、の順に積み上げると全段が中間状態として運用可能。つまり Phase 計画の最終到達点をここに置き直せる。
