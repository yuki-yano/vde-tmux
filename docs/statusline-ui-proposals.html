vde-tmux statusline
表示形式の再設計 4案

現行実装(statusline-sessions / statusline-category / statusline-agent-badge の3コマンド構成)の調査と、実運用スクリーンショット(vde-tmux-manager ベースの pill スタイル)に基づき、tmux ステータスバーの役割定義から表示・運用・アーキテクチャまでを見直す。全案が「アイコン pill のカテゴリ + index 付きセッション + current pill」という現行のビジュアル言語を継承し、その上にサイドバー再設計(Plan 13〜17)と共有する状態語彙(▲●✓○)を重ねる。

役割定義 現状と課題 共通前提 案A グリフ注入 案B 状態pill 案C 例外チャネル 案D カテゴリ集約 アーキテクチャ 修正・機能表 推奨

00ステータスバーはどうあるべきか

サイドバー再設計後の世界では、両者の役割分担を先に決めないと表示項目が決められない。結論から言うと:

サイドバー(Plan 13〜17)ステータスバー
性格triage と操作の場。開いているときだけ存在常時視界にある1行。閉じられない・操作は最小
答える問い「どれから対応するか」「今どこまで進んだか」「今いる場所はどこか」「視界の外で何かが自分を待っていないか」
情報の深さagent 単位、prompt・進捗・LIVE まで集約値と例外のみ。詳細はサイドバー/jump に委譲
面積幅40列 × 全高数十セル。1セルが貴重

つまりステータスバーの本質は「現在地の表示」と「不在検知の安全網(サイドバーを閉じている・別 session にいるときに attention を取りこぼさない)」の2つで、それ以外は全部ノイズ候補。現行の3コマンドをこの物差しで評価し直すのが本提案の軸になる。

01現状の表示と課題

実運用(vde-tmux-manager + dotfiles)のスクリーンショットを再現したもの。カテゴリは display_names でアイコン化された pill(アクティブ = オレンジ)、続いて現 category のセッションが index 付きで並び、current session がラベンダーの pill。右端に window pill。

🏠 🌐 1 duel-logger 2 opener-rate 3 vde-monitor 4 vde-tmux 1 node
現行の実運用。pill の見た目は dotfiles 側の prefix/suffix/colors 設定 + display_names(アイコン)で実現されている
🔴 1:main 2:sub 1:nvim 2:zsh* private work running:2
参考: vde-tmux(新実装)を素の config で使った場合の出力。pill も色も付かない

つまり「見た目の骨格」は既に dotfiles 側で完成しており、vde-tmux 新実装はそれを #() の呼び先差し替えで再現できる状態にある(migration.md の M7)。本提案の主眼は pill の骨格を壊さずに「agent の状態」をこのバーに正しく載せることになる。

調査で確認した課題

  • 情報の重複 — session badge(🔴 = blocked あり)と running:2(agent-badge)は同じ「agent の状態」を別の粒度・別の記法で二重表示している。しかも後者は英語ラベルで語彙が揃っていない。
  • category に件数がない — private work と名前を列挙するだけで、「そこに何があるか」を何も伝えない。現在地表示としても弱い(active 色も既定では付かない)。
  • current session が既定で無区別 — current/other のスタイル既定が同一。バッジ以外にカレントを示すものがない。
  • 絵文字幅問題 — 🔴🟡 の幅揺れを suffix: " " で帳尻合わせしている(plan-09 に対症療法と明記あり)。サイドバー(Plan 13)は ▲●✓○ に移行するため、放置すると語彙が分裂する。
  • dead config — statusline.agent_badge.enabled はどこからも読まれていない(README には設定例として載っている)。SegmentColors.outer_bg も定義のみで未配線。
  • stale バッジ — daemon がクラッシュすると @vde_session_status に最後の値が残り続け、「生きているように見える」偽表示になる。
  • 更新遅延が暗黙前提 — 反映は daemon poll(1s)+ tmux status-interval の合成。status-interval の推奨値がどこにも書かれていない。
  • クリック非対応 — switch <n> サブコマンドはあるが #[range=...] を生成しないため、クリック切替は利用者の自作頼み。

02共通前提 — 現行ビジュアルの継承とサイドバーとの語彙統一

4案すべてに共通する前提。個別の案を選ぶ前に、これだけは確定させるべきもの。

  • pill の骨格は壊さない: 「カテゴリ = アイコン pill」「セッション = index + 名前」「current = 塗り pill」「window = 右端 pill」という現行の文法は維持する。提案が加えるのは状態(▲●✓○)のレイヤーだけ。pill 装飾は引き続き dotfiles(prefix/suffix/colors)の管轄とし、vde-tmux 側は素材(プレースホルダと状態値)を供給する。
  • グリフと色の統一: ▲(blocked・赤)/ ●(working・緑)/ ✓(done・シアン)/ ○(idle・無彩)。badge.glyphs は sidebar と共有の top-level 設定なので、Plan 13 の既定値変更で自動的に単幅グリフ化されるのは session badge のみ。agent-badge(running:N)の summary 置換と suffix: " " の既定変更は明示作業として必要。
  • 件数表記の統一: サイドバーのフィルタバー(Plan 17)・rail と同じ ▲2 ●1 形式。running:2 表記は廃止。
  • agent-badge コマンドの再定義: 現行 vt statusline-agent-badge(running:2)を、状態別カウント ▲2 ●1(0件は省略、tmux 色マークアップ付き)を出す vt statusline summary に置き換える。dead config の agent_badge.enabled はこのタイミングで整理。
  • current session の既定スタイル: config 未設定でも bold + 色が付く既定に変える(現行は「設定しなければ全部同じ見た目」)。

サイドバーとの併用整合 — pill とフラットは共存できる

ステータスバーは「クローム」(常設の装飾面)、サイドバーは「コンテンツペイン」(密度優先の作業面)であり、質感が違うこと自体は正しい(VS Code のアクティビティバーとエクスプローラの関係と同じ)。両者を繋ぐのは見た目の統一ではなく語彙の統一で、次の4ルールを守れば違和感は出ない:

  • 状態4色(赤・緑・シアン・無彩)は状態専用に予約。装飾 pill はラベンダー・オレンジ・クリーム等、状態色と被らない色相に留める(現行配色は既に満たしている)。
  • 塗り背景の意味は「現在地/選択」と「警報(赤のみ)」の2つに限定。バーの current pill ↔ サイドバーの選択行 bg、案Cの赤 pill ↔ TRIAGE の赤見出しが、それぞれ同じ文法として対応する。
  • グリフは両面で同一セット・同一色(badge.glyphs 共有で自動担保)。
  • 状態グリフは pill と一体化させ、浮かせない。裸のグリフを pill の外に置くと所属が曖昧になり pill 列のリズムから浮く。かといって状態色をそのまま装飾背景に直置きするとコントラストが崩れる。解は2つ:
    (a) 内側 + 色バリアント(推奨・既定) — pill 内にグリフを置き、背景の明度に応じた濃色/明色バリアントを使う(▲ 4 vde-tmux)。pill の一体感を保ち、実装も最軽量
    (b) 分割コンパートメント(opt-in) — pill 左端にバッジ専用の状態色区画を融合(▲4 vde-tmux)。powerline 的でより主張が強い
    非 pill 要素(通常セッションの ●2 opener-rate)は従来どおり裸のグリフ前置でよい — 地の上なら浮かない。

この観点で唯一要注意なのが案B(State Pills): pill の塗りを状態色にするため「塗り=現在地」の文法が崩れ、装飾 pill と意味が衝突する。案Bを opt-in に留める理由はここにある。

AGlyph Injection — 現行 pill に状態グリフを注入

向くケース:今の見た目をほぼ変えたくない / まず最小の手数で agent 状態をバーに載せたい

🏠 ▲ 🌐 ○1 duel-logger ●2 opener-rate ✓3 vde-monitor ▲ 4 vde-tmux ▲1 ●1 ✓1 ○1 1 node
推奨配置(内側 + 色バリアント): グリフは pill 内、明るい pill には濃色バリアント。非 pill のセッションは裸のグリフ前置のまま
🏠 ▲🌐 ○1 duel-logger ●2 opener-rate ✓3 vde-monitor ▲4 vde-tmux ▲1 ●1 ✓1 ○1 1 node
代替配置(分割コンパートメント): pill 左端に状態色の小区画を融合。より主張が強い opt-in
🏠 ▲🌐 1 duel-logger … ▲4 vde-tmux (参考・不採用)
不採用(外側の裸グリフ): 所属が曖昧で pill 列から浮く
  • セッションへのグリフ前置は実は実装済み(session badge の前置、`{badge}{label}`)。Plan 13 の badge.glyphs 既定変更で 🔴→▲ になり、単幅化で幅ズレも消える。idle(○)も既定で表示する — これにより無印の意味が「agent なし」だけに純化され、「idle の agent が居る」と「agent が居ない」が一目で区別できる(agent の居ない session は現実装でも daemon がバッジを書かないため無印)。静かにしたい人向けに hide_idle_badge: true オプションを用意(既定 false)。
  • pill 付き要素(current session・カテゴリ)のバッジは内側配置: グリフを pill 内に置き、pill 背景の明度に応じた状態色バリアント(ラベンダー上は深紅等)を使う(併用整合ルール4)。renderer が badge 部分に個別 fg(#[fg=<state_variant>]▲#[fg=<label_fg>]、bg は pill のまま)を出す badge_style: inline | compartment | plain の新設が必要(既定 inline)。compartment(左端の状態色区画)はより主張の強い opt-in。
  • カテゴリの最悪状態グリフ: category format に {badge} プレースホルダを追加し、そのカテゴリ所属 session の badge の min(最悪値)を注入(新実装)。非アクティブカテゴリ(🏠)にも ▲ が付けば「別カテゴリで待たれている」が icon レベルで分かる。
  • 右端に summary(▲1 ●1、0件省略)。running:2 は廃止。

Pros

  • dotfiles の pill 設定を1文字も変えずに移行できる(グリフは既存プレースホルダに乗る)
  • カテゴリ icon への状態伝播で「視界の外」がカテゴリ粒度まで見える

Cons

  • 「どの session が待っているか」は現 category 外だと分からない(カテゴリ粒度止まり)
  • グリフの分だけ左側が最大4〜5セル伸びる
実装: badge_style(inline)+ hide_idle_badge + summary コマンド + dead config 配線で規模感 中。category への {badge} はグリフ→BadgeState の逆引きか構造化経路が必要(中〜大)なため確定計画では Step 5 送り(§05 参照)。

BState Pills — pill の色そのものが状態を語る

向くケース:グリフの文字を増やしたくない / pill の色彩をもう一段活用したい / 一瞥での判別速度を最優先

🏠 🌐 1 duel-logger 2 opener-rate 3 vde-monitor 4 vde-tmux 1 node
全セッションを pill 化し、背景色 = 状態(赤=blocked / 緑=working / シアン=done / 暗=idle)。current は太字+明度で区別
  • セッションを全て pill にし、pill の背景色を badge 状態に連動させる: blocked=赤 / working=緑 / done=シアン / idle=暗色。current session は同色系の高明度 + 太字。
  • グリフ文字を足さないため幅の増加はゼロ。色だけで「どの session が待っているか」が icon 的に分かる。
  • 実現には vde-tmux 側に 状態別カラーセット(sessions.state_colors.{blocked,working,done,idle} で fg/bg を指定し、`{session}` セグメントのスタイル解決時に badge 状態で切り替える)の新設が必要。pill の丸め(prefix/suffix)は従来どおり dotfiles。

Pros

  • 判別速度が最速(色面 > グリフ > テキスト)。幅コストゼロ
  • 「バーの中に rail がある」状態になり、サイドバー rail と完全に同じ語彙

Cons

  • 色覚特性・配色テーマとの相性問題をグリフなしで受けることになる(A のグリフ併用で緩和可能)
  • state_colors の新設 + スタイル解決の条件分岐で、4案中では statusline 側の改修が最大
  • current の区別が「明度差」になり、赤 pill(blocked)と赤系 current の衝突に配色設計が要る
実装: SegmentStyle の状態別解決(state_colors)+ current との合成規則。規模感 中。

CException Channel — 右端に「よそで待たれているもの」だけ

向くケース:サイドバー(TRIAGE)主力で、status bar は不在検知の安全網に徹させたい / 左側は現行のまま一切触りたくない

🏠 🌐 1 duel-logger 2 opener-rate 3 vde-monitor 4 vde-tmux ▲ llm-proxy · perm 2m ●2 1 node
左は現行のまま。右に「現在 attach していない session の blocked」を赤 pill で名指し + 残りは集約
🏠 🌐 1 duel-logger 2 opener-rate 3 vde-monitor 4 vde-tmux ●3 ✓1 1 node
例外が無いとき: 右は静かな集約だけに縮む
  • 左側(カテゴリ pill + セッション列挙)は現行のまま。右端に vt statusline attention を追加するだけの増築案。
  • 出力は「現在 attach していない session の blocked」を ▲ {session} · {reason} {経過} で名指し(最も古い1件 + +N)、それ以外は集約値(●2 ✓1)。blocked が無ければ集約のみ。
  • 「見えている場所の問題は報せない。見えていない場所の問題だけ報せる」— サイドバーの TRIAGE(スコープ貫通)と対になる通知設計。
  • データは daemon Query(wait_reason・経過は daemon が持つ)+ agent-badge と同じ list-panes フォールバックの2段構え。

Pros

  • 平常時は最小、異常時だけ具体的 — 注意資源の使い方が最も正しい
  • 左側の運用(index 切替・pill 装飾)に一切触れない純増築
  • Plan 15(TRIAGE)完了後なら daemon 側に blocked 集合が既にあり実装が薄い

Cons

  • 右端の幅が動的に変わり、window pill の位置がガタつく(固定幅確保オプションで緩和)
  • daemon 停止時は例外チャネルが沈黙する(heartbeat での stale 表示が前提)
実装: vt statusline attention 新設のみ。規模感 小〜中(daemon Query 拡張含む)。

DCategory Dashboard — アイコン pill を集約器にする

向くケース:session 数が増えて列挙が window リストを圧迫し始めた / カテゴリ単位の把握が主で session 名は current だけ見えればよい

🏠 2 ▲ 🌐 4 ▲ 4 vde-tmux 1:nvim 2:zsh* 3:codex ▲1 ●2 1 node
カテゴリ pill = 内側バッジ(最悪状態)+ アイコン + session 数。セッション列挙をやめ current pill だけ残す

  • current session は pill 1つだけ残す(現在地表示)。session 単位の把握と切替は session-manager popup / サイドバーに委譲。
  • index 切替は「カテゴリ切替(statusline-category switch)+ popup」の2段になる。index 直打ち切替の運用を手放せるかがこの案の分水嶺。

  • session 数に対して幅が一定(カテゴリ数にしか比例しない)。window リストに最大面積
  • カテゴリ横断の状態が icon 列で一望でき、B/C の要素(色・例外)とも併用可能

Cons

  • index 即切替の筋肉記憶を壊す(最大の代償)
  • 「どの session か」の解決に必ず1ホップ要る
実装: category format への {count}/{badge} 追加(案Aと共通)+ sessions の current-only モード。規模感 小〜中。

03運用アーキテクチャ — #() 継続か option bus か

表示案と独立に決められる、更新の仕組みの選択。

現行: #(vt ...) コマンド置換提案: option bus 配信
仕組みstatus-interval ごとに tmux が vt を fork して stdout を埋め込むdaemon が描画済み文字列を @vde_statusline_left/right に書き、tmux.conf は #{E:@vde_statusline_left} を参照
更新コストinterval ごとにプロセス起動 ×コマンド数(sessions + category + summary で3 fork/interval)daemon の poll 内で差分時のみ set-option。fork ゼロ
反映遅延poll_ms + status-intervalpoll_ms + status-interval(同じだが fork 分の揺らぎがない)
daemon 停止時sessions/category は素で動く(badge のみ stale)文字列全体が stale — heartbeat が必須(下記)
実装の場所CLI(現行のまま)daemon の effect に SetStatusline を追加。session badge と同じ経路(@vde_session_status の前例あり)

stale 対策(どちらの構成でも入れるべき): daemon が poll ごとに @vde_heartbeat(epoch)を書き、描画側は heartbeat が5秒より古ければバッジ・サマリを dim の ? に落とす。「daemon が死んでいるのに元気そうに見える」現行の偽表示を潰す。加えて shutdown 時の全消去(既存)はそのまま。

推奨は「当面 #() を維持し、Plan 17 完了後に option bus へ移行」。理由: option bus は daemon への一極化を進める設計で、statusline の daemon 依存を高める。sessions/category が daemon 無しで動く現行の頑健さは、daemon 自体の安定運用が実証されるまで保険として残す価値がある。status-interval は 1 推奨を README に明記する(現状どこにも書かれていない)。

05推奨と進め方

推奨: 案A(グリフ注入)+ 案C(例外チャネル)の組み合わせを既定とし、案B(state pill)は state_colors の opt-in 機能として提供する。A は現行の pill 骨格・index 切替・dotfiles 設定を一切壊さず、Plan 13 のグリフ統一にほぼ相乗りできる。C は左側に触れない純増築で、サイドバー TRIAGE の対になる。B は判別速度で最強だが改修が最大なので、A+C 運用後に「グリフでは物足りない」と感じた場合の強化として位置づける。D(カテゴリ集約)は session 数が列挙に耐えなくなったときの縮退先として config で選べるようにしておく。

🏠 ▲ 🌐 ○1 duel-logger ●2 opener-rate ✓3 vde-monitor ▲ 4 vde-tmux ▲ llm-proxy · perm 2m ●2 1 node
