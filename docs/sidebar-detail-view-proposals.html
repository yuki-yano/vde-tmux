展開ビューの改善提案

実スクリーンショット(2026-07-06)の表示状態をもとに、fisheye 展開時の情報構成を再検討する。 指摘の2点(status 行の状態色/chat 行と展開行の情報重複)に加え、時間表記とメタ行の冗長さも対象にする。

  • ② status 行が無彩色 — 状態は本来このサイドバーの主役なのに、展開の中では色の文法(▲赤/●緑/✓シアン/○灰)が途切れる

Item 1

chat 行と展開行の重複解消

展開中の chat 行のラベル(truncate された prompt)と、展開1行目の prompt 全文が重複している。

案 A — 展開中は chat 行を agent 名だけに縮約

   ▾ ○ claude
     M-Eは対応させなくて良いから、M-eのmapだけ差し替えといて
     …
   ▸ ○ claude: plan-12レビュー結果を確認し…  38h
  • 重複が消え、prompt は展開1行目(全文)が唯一の置き場になる
  • chat 行の右ラベル(経過時間)も展開中は非表示にでき、メタ行(Item 3)と役割が分離する
  • 折りたたみ時は従来どおり(truncate prompt + 右ラベル)なので一覧性は落ちない
  • 展開/折りたたみで行の見た目が変わる(識別は agent 名とバッジで保たれる)

案 B — 展開1行目の prompt 行を廃止

   ▾ ○ claude: M-Eは対応させなくて良いから、M-eのma…
     status: idle …
  • 展開の行数が 1 行減る
  • truncate の続き(prompt 全文)をサイドバー内で読む手段が無くなり、preview 頼みになる
  • 長い prompt ほど情報が失われる、逆方向の割り切り

案 C — chat 行を維持し prompt 行は続きだけ

   ▾ ○ claude: M-Eは対応させなくて良いから、M-eのma…
     …pだけ差し替えといて
  • 重複ゼロで全文も読める
  • truncate 位置が幅依存なので「続き」の切り出しが脆く、折り返し表示に近い複雑さになる
  • 2 行に分かれた文はどのみち読みにくい

推奨: 案 A。「折りたたみ = ダイジェスト 1 行、展開 = 全文+詳細」という fisheye の意味論に一番素直。 展開中の右ラベル非表示までセットで入れると、経過時間の置き場も Item 3 のメタ行に一本化される。

実装影響: tree.rs push_chat_row で expanded のとき label を agent 名のみにする。render.rs right_label は row.expanded && kind==Chat で None。

Item 2

status 行の状態色

status の値は 4 値バッジと同じ語彙なのに、展開の中では無彩色のテキストになっている。

案 A — グリフ+値だけ状態色

     ● running · 12m
     ▲ permission (approve request?)
     ○ idle
  • バッジと同じグリフ+色で「状態の語彙」が展開内でも一貫する
  • status: ラベル自体を省ける(グリフがあれば状態行だと自明)
  • wait_reason は括弧のまま detail 色で従属情報に留める
  • ラベル省略に伴い、既存の「status: 」表記に慣れた目には一瞬新しい

案 B — ラベル維持、値のみ色付け

     status: ○ idle
     status: ● running
  • 最小差分で指摘に応える保守的な形
  • status: の 8 桁が毎回先頭に居座り、幅 40 では値の余白を圧迫する

案 C — 行全体を状態色で塗る

     status: running (build 実行中)
  • 最も目立つ
  • 「状態色はバッジと右ラベルだけ、本文は通常色」という多トーン方針(理想形で確定済み)を壊す
  • 展開が多いと画面が色で騒がしくなる

推奨: 案 A。色はグリフ+状態語だけに載せ、補足(wait_reason)は detail 色に残す。 確定済みの多トーン方針と整合し、Item 3 の統合メタ行にそのまま組み込める。

実装影響: detail 行は既に rollup / badge_state を保持しているため、render.rs で id が detail::…::status の行だけ「グリフ+状態語 span を状態色、残りを detail 色」に分ければよい(daemon 変更不要)。

Item 3

メタ行の圧縮と時間表記

status / elapsed / session の 3 行はラベルが冗長で、時間は分単位の生値。 さらに idle では「起動からの経過」より「いつ完了したか」の方が有用。

案 A — 状態行+場所行の 2 行に統合

  running の場合
     ● running · 12m
     vde-tmux · %51
  idle の場合(completed_at ベース)
     ○ idle · done 38h ago
     vde-tmux · %55
  • 3 行 → 2 行。ラベル(status:/elapsed:/session:)を全廃し値だけにする
  • 時間の意味を状態に合わせて切替: running=経過、idle/done=完了からの時間、blocked=待ち時間。idle の「2279m04s」問題が本質から消える
  • 時間は humanize(45s / 12m / 1h30m / 38h / 2d)。未展開行の右ラベルにも同じ関数を使う
  • completed_at を RowMeta に載せる daemon 側変更が必要

案 B — 3 行維持、値だけ改善

     status: ○ idle
     elapsed: 38h00m
     session: vde-tmux / pane: %55
  • 構造を変えないので実装が最小(humanize + 色のみ)
  • 縦 3 行の消費と session 行の冗長さは残る
  • idle の elapsed が「起動からの経過」のままで、値の有用性が低い

案 C — 1 行に全部凝縮

     ○ idle · done 38h ago · vde-tmux · %55
  • 展開が最小 3 行(prompt+メタ+アクション)になり、多数展開時に強い
  • 幅 40 で truncate されやすく、後半(場所情報)が欠けがち
  • pin 時の要約 1 行(meta 行)と見た目が紛らわしい

推奨: 案 A。「状態の行(色付き)」と「場所の行(灰色)」の 2 行構成は、Item 2 の色付けと自然に合成できる。 session 名が repo 行と同じ場合は場所行を %55 だけに省略する最適化も入れられる(初期実装では常時表示でよい)。

実装影響: tree.rs push_chat_detail_rows の再構成(status/elapsed/session の 3 detail 行 → state 行+place 行)、PaneSnapshot.completed_at(既存)を AgentPane/RowMeta に配線、elapsed_label の humanize 化(右ラベル・TRIAGE・meta 行と共通化)。

Item 5

色システム全体の再検討

機能追加のたびに色が増えてきたので、現状の色マップを棚卸しして役割を再定義する。

現状の色マップ(監査)

案 A — 5 族の文法を明文化+衝突だけ解消

  状態族   ▲ ● ✓ ○  (バッジ・右ラベル・状態行・ヘッダー)
  構造族   repo ◆ category main+2  (branch は 73 へ降格)
  操作族   ✦ ≣ ▎ ⌕  (pin / mode / active / preview = ラベンダー)
  本文族   本文=通常色 / 補足=246 / 記号=暗灰
  実況     LIVE  (マゼンタは LIVE 専用の孤立色として維持)
  • 変更は実質 branch の既定色を明シアン → 淡シアン(73)へ落とすだけ。✓ done がシアンの唯一の「強い」使い手になり、branch は jump と同じ「場所・移動」族に収まる
  • 既にできている秩序(状態 4 色/構造 2 色/操作ラベンダー)を規約として README・theme コメントに固定し、今後の機能追加で色が増殖するのを防ぐ
  • 全キーが config 化済みなので、既定値の変更のみで互換の心配がない
  • branch の視認性は今より一段下がる(構造情報としては適正、という判断)

案 B — 構造情報をモノトーン化(状態だけが色)

  ◆ public ──────────
    vde-tmux  main +2
      ▸ ○ claude: plan-12…  38h
  • 色=状態、と意味が一対一になり、「赤いものだけ見ればいい」が極まる
  • category / repo の視覚的区別(Item 2 で入れたばかりの 3 役 3 色)を放棄することになる
  • statusline のピーチピルとの一貫性が崩れ、ツール全体では逆に不揃いになる

案 C — アクセントをラベンダー 1 系統に統合

  ◆ public
    vde-tmux  main
      LIVE tail · %51
  • 状態 4 色+ラベンダー濃淡だけの 2 系統になり、画面全体の彩度が下がって落ち着く
  • repo と category の区別が濃淡頼みになり、Item 2 の解決を半分巻き戻す
  • 「ラベンダー=操作・現在地」の意味が薄まる(構造にも実況にも使うため)

推奨: 案 A。現状は偶発的にせよ族がほぼ成立しており、壊れているのは「✓ とbranch のシアン衝突」1 点だけ。 branch を 73 に落とし、5 族(状態/構造/操作/本文/実況)を規約として明文化するのが最小コストで最大の整合。 LIVE のマゼンタは「画面内で唯一の実況」という特異性の表現として孤立を許容する。

実装影響: SidebarRenderTheme::default().branch を Color::Cyan → Color::Indexed(73) に変更(config `colors.branch` で従来色に戻せる)。加えて theme 構造体コメントと README に 5 族の規約を記載。
